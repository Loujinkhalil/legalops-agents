To consider later — identified issues and deficiencies
This section records issues identified during the workflow analysis that require attention. It is updated as new issues are found. No action is taken on these until the full workflow analysis is complete.

Issue 1 — Layer 0 is defined but empty
Identified: Phase 1 vs. skills comparison
Source: rpd-decision-analysis skill Phases 1 and 2
Layer 0 is defined as "system prerequisite: must be confirmed before any agent activates" but no agents are assigned to it. The rpd-decision-analysis skill requires two complete phases before the decision is opened: Phase 1 (triage — case strength assessment and claim type identification) and Phase 2 (document review — BOC narrative, supporting documents, hearing transcript, post-hearing submissions). Neither phase is represented in the agent layer system. The 15-technique system assumes this work is done but does not formalize it as agents.
Effect: The agent system has no documented starting point before T1. A practitioner or agent using the layer model as a workflow would open the RPD decision without first reviewing the BOC narrative and supporting documents — violating the foundational rule of the rpd-decision-analysis skill that evidence must be known before the decision is read.

Issue 2 — No segmentation agent for multi-session work
Identified: Phase 1 vs. skills comparison
Source: memory-aware-analysis-v2 skill Phase 0
The memory-aware-analysis-v2 skill requires Phase 0 (segmentation — building the extraction file and segment map) before any pass begins on a document too large for a single session. RPD decisions and transcripts routinely exceed single-session context limits. No agent or layer in the system represents this segmentation step.
Effect: For multi-session files, the agent system provides no architecture for dividing the decision into processable segments, building the extraction file, or managing session handoffs. The system works as designed for single-session files but has no protocol for the majority of real cases.

Issue 3 — ndp-citation skill is not connected to the agent system
Identified: Phase 1 vs. skills comparison
Source: ndp-citation skill
T14 verifies the NDP version number from the disclosure document list but does not fetch NDP content. T10 flags non-engagement with NDP tabs but cannot confirm what a tab says without the tab content being in the session. The ndp-citation skill is the tool that fetches and extracts NDP tab content — Mode A for RPD-cited tabs and Mode B for issue-based retrieval. No agent in the system calls it.
Effect: T10 Cepeda-Gutierrez arguments are structural — they identify that a tab should have been cited — but cannot confirm what the tab would have shown. NDP content retrieval is a prerequisite for building the affirmative NDP argument in the RAD memorandum, which the agent system does not currently support.

Issue 4 — irb-guideline-caselaw skill is not connected to the agent system
Identified: Phase 1 vs. skills comparison
Source: irb-guideline-caselaw skill
T1 extracts guideline requirements from the decision text as written by the RPD. It does not fetch the actual guideline text or retrieve the supporting caselaw that defines what the guideline requires. The irb-guideline-caselaw skill performs that retrieval and produces a structured caselaw log organized by proposition. Without it, T1 and T13 operate from Claude's general knowledge of Guideline 9, not from the verified guideline text and its cited authorities.
Effect: T1 compliance benchmarks and T13 behavioral expectation findings may be grounded in Claude's recollection of the guideline rather than its verified text. On appeal, arguments built on imprecise guideline language are vulnerable. The irb-guideline-caselaw output is also the source of the caselaw stacking for the RAD memorandum's guideline violation section — this is currently unconnected.

Issue 5 — the 15-technique system is a skill-based behavioral framework — all six architecture conditions are behavioral approximations only
Identified: Reading of "Stateless by Design" paper (paper_complete_v4_pass2_inprogress.md) — sections 4.5, 5.1, 5.2, 5.3
Source: Paper section 4.5 (evaluation of skill-based behavioral frameworks); section 5.1 (framework registry); section 5.2 (pass schedule); section 5.3 (pass-dimension matrix)
The paper evaluates five categories of existing approaches against six architecture conditions. The 15-technique RPD system falls into category 4.5 — skill-based behavioral frameworks — which the paper finds provides only behavioral approximations of all six conditions, with compliance-dependent failure modes throughout.
The six conditions and how the 15-technique system fails each one structurally:
C1 — session statelessness: the techniques produce output objects but define no session handoff obligation. Every Reading 1 object, Reading 2 object, clarification object, and misunderstanding object lives in in-context memory. When the session ends, all of it is gone. Memory-aware-analysis-v2 provides session persistence but the techniques do not call it as an architectural commitment — they produce outputs and leave externalization to deployment choice.
C2 — researcher-driven corpus growth: the techniques assume a fixed document set. There is no retrieval queue, no pending set, no citation following mechanism, no query generation mechanism. When T1 identifies Guideline 9 or T10 identifies a missing NDP tab, nothing in the system generates a retrieval action to fetch those documents.
C3 — task model and coverage tracking: there is no task model with requirement elements and predicate questions. Coverage state transitions — whether a legal principle has been satisfied by the evidence — are determined by Claude's judgment applied to prose, not by a computable predicate applied to a defined data structure. Two sessions reading the same accumulation file may reach different conclusions about whether an element is closed.
A1 — framework scheduling: the pass schedule is implicit and positional. Analysis proceeds in whatever order the session processes the document. There is no framework registry defining which frameworks apply to which document segments, no pass schedule generated from coverage requirements, no enforcement that the IRB credibility guidelines were loaded alongside the paragraph being analyzed.
A2 — dimension coverage tracking: there is no pass-dimension matrix. A paragraph that was analyzed with the NDP framework absent is indistinguishable from a paragraph that was analyzed with it present and produced no finding. The absence of a T10 flag for a given paragraph does not mean the NDP framework was applied and found nothing — it may mean it was never applied. This is the failure mode the paper names dimension blindness.
A3 — cross-pass finding integration: there is no defined integration operation. If T5 and T13 both run on the same exchange and produce findings that interact — a misread answer that is also a behavioral expectation violation — there is no architectural step that reads both findings together to detect the compound error. Integration is left to the model's general reading of whatever is currently in context.
The root cause — identified in the paper:
The framework registry is a human-defined structure. The researcher provides it before any pass begins. It contains every framework required to analyze the document, with applicability conditions per framework and load priority. An agent cannot construct the framework registry. Without it, the agent cannot know which frameworks to load in which pass, cannot generate a coverage-driven pass schedule, and cannot initialize the pass-dimension matrix. The 15-technique system has no framework registry, no applicability conditions, no pass schedule, and no matrix. All six conditions therefore remain behavioral approximations dependent on model compliance rather than structural enforcement.
What must be added:
A pre-analysis human-defined step that produces a framework registry for each file type — RPD decision, BOC narrative, hearing transcript — before any technique activates. The registry must contain: framework identifier, description, framework type (analytical framework or evidentiary comparator), applicability condition (which document segment types trigger this framework), and load priority. This registry is written to external memory at task initiation and governs the pass schedule for the entire analysis. It is the foundation the 15 techniques currently lack.

Issue 6 — legal principles are not converted into predicate questions — coverage tracking is open-ended prose judgment
Identified: Reading of "Stateless by Design" paper — section 5.6 (task model schema) and section 5.8 (coverage state schema)
Source: Paper sections 5.6, 5.7, 5.8
The paper specifies that each requirement element — each legal principle extracted from a governing framework — must be converted into a single directly answerable yes-or-no predicate question. This is the evaluation interface. Coverage state transitions (open → addressed → closed) are determined by applying the predicate question to structured evidence summaries, not by Claude's open-ended reading of accumulated prose.
In the RPD context, examples of what predicate questions should look like:

From Guideline 9 § 6.1: "Did the RPD apply a behavioral expectation to this claimant that would not apply to a heterosexual claimant in the same country and circumstances?"
From IRB credibility assessment: "Did the RPD draw an adverse inference from an omission without first confirming the subject was put to the claimant at the hearing?"
From Cepeda-Gutierrez: "Did the RPD fail to cite a directly relevant NDP tab on a core adverse finding?"

Each question is answerable yes or no from the evidence. An element is closed when the analyzed set contains sufficient evidence to answer its predicate question yes. It reopens if later analysis contradicts prior satisfying evidence.
What the 15-technique system does instead: The techniques produce output objects — reading objects, clarification objects, expectation objects — but these objects are not evaluated against predicate questions. Whether a given legal principle has been satisfied or violated is left to Claude's in-context judgment at the time of analysis. Two sessions analyzing the same paragraph may reach different conclusions. There is no defined question, no structured input, and no computable stopping condition.
Effect: Coverage tracking is behavioral and interpretively variable. The same gap the paper identifies for skill-based behavioral frameworks in section 4.5 applies directly: "coverage state transitions are determined by the model's judgment applied to prose in an accumulation file or instruction set. The outcome is interpretively variable: different sessions reading the same prose may reach different conclusions about whether an element's satisfaction condition has been met."
What must be added: For each governing framework in the registry, a task model entry for every extractable legal principle, with a single predicate question per principle. The task model is written to external memory before analysis begins. Output objects from the techniques are structured as answers to these predicate questions — not as general findings. Coverage state transitions follow defined rules applied to the structured answers.

Issue 7 — the transcript is used as a comparator but never formally analyzed as a corpus document — the dual-role problem
Identified: Reading of "Stateless by Design" paper — section 5.9 (state validity and document-level gate), section 5.11 (what the architecture provides)
Source: Paper sections 5.9, 5.11
The paper identifies that an evidentiary comparator can appear in two roles simultaneously: as a reference framework loaded alongside the subject document in a pass, and as a corpus document in its own right that must be formally analyzed and whose findings must satisfy dimension coverage completeness before they can advance the corpus-level coverage state.
In the RPD system, the hearing transcript occupies both roles:

As a reference framework: it is loaded alongside the RPD decision in passes for T5 (member heard vs. understood), T11 (clarification sequence), and T12 (member's own questions), where its content is compared against the decision's account of what was said
As a corpus document: it contains its own evidence relevant to every requirement element — positive evidence of genuine experience, representation failure indicators, clarification events — that must be systematically extracted and recorded in the analyzed set

The current system uses the transcript only in the first role. T5, T11, and T12 are activated by triggers in the RPD decision and then look into the transcript for specific exchanges. The transcript is never formally analyzed as a corpus document through a framework registry. Its findings are captured in technique output objects but are never entered into an analyzed set, never evaluated against predicate questions, and never subjected to a dimension coverage completeness condition before being used.
Effect: The transcript's evidentiary contribution to the task model is incomplete and unstructured. Positive evidence extracted in T6 comes from the transcript but is never systematically verified against all applicable frameworks. Representation failure indicators in T15 depend on the transcript but the transcript itself has no formal processing status in any queue. The document-level gate — which requires dimension coverage completeness before a document's findings can advance coverage state — is never applied to the transcript.
What must be added: The transcript must be formally registered in the framework registry as an evidentiary comparator and simultaneously placed in the pending set of the retrieval queue as a corpus document requiring its own formal analysis passes. Its processing status must reach complete — meaning its own pass-dimension matrix satisfies dimension coverage completeness against its applicable frameworks — before its findings can contribute to corpus-level coverage state transitions.

Issue 8 — five required human involvement points are undefined in the current system
Identified: Reading of "Stateless by Design" paper — section 6.4 (human-researcher integration)
Source: Paper section 6.4
The paper identifies five points where human involvement is architecturally required — not optional, not a deployment choice. An agent cannot perform these functions. The 15-technique RPD system has none of them defined.
The five required human involvement points, translated directly to the RPD context:

Framework registry definition — the legal practitioner must define which frameworks apply to the RPD decision before analysis begins: Guideline 9, IRB credibility assessment guide, IRB weighing evidence guide, NDP country evidence, statutory definition of protected person, transcript. Claude cannot determine this. This is a legal judgment requiring domain knowledge.
Cross-pass integration review — after all applicable framework passes are complete for a given RPD paragraph, the practitioner must confirm that the combination of findings from all frameworks applied to that paragraph has been read together and any compound errors identified. Claude can load the findings and flag combinations but cannot determine whether integration is analytically complete.
Task model definition — the practitioner must write the predicate questions for every legal principle extracted from every governing framework. Claude cannot construct the task model. The decomposition of legal principles into tractable yes-or-no questions requires legal judgment.
Coverage completeness assessment — when every predicate question has been answered yes, the practitioner must assess whether the quality of the satisfying evidence meets the standard required for the RAD memorandum. Structural coverage completeness is a necessary but not sufficient condition for a defensible submission.
Task model expansion — when analysis reveals a legal issue not anticipated in the initial task model, the practitioner must decide whether to add a new requirement element. Claude can flag unanticipated issues but cannot determine whether they rise to the level of a ground of appeal.

Effect: Without defined involvement points, the system has no architecture for managing the boundary between what Claude does and what the practitioner must do. In practice, Claude is currently expected to perform functions the paper establishes cannot be automated — including framework registry construction, task model definition, and coverage completeness assessment. This produces outputs that are analytically incomplete in ways neither Claude nor the practitioner can detect.

Issue 9 — pre-applicability failure is undetectable and will remain so even after a framework registry is added
Identified: Reading of "Stateless by Design" paper — section 6.2 limitation 5 (framework applicability problem)
Source: Paper section 6.2, limitation 5
The paper distinguishes two failure modes. Dimension blindness occurs when a pending cell — a (segment, framework) pairing correctly entered in the matrix — is never processed. This is detectable: the matrix shows the cell as pending. Pre-applicability failure occurs when a framework-segment pairing that should have been entered as pending was never entered at all, because the applicability condition was written too narrowly before the document was read. The matrix cannot flag a missing cell. The coverage requirement applies only to cells initialized as pending; a cell never initialized is invisible to the coverage check.
In the RPD context: if the applicability condition for the IRB weighing evidence guide is written as "applies to paragraphs explicitly assessing documentary evidence weight," it will miss paragraphs that implicitly apply weighing rules without naming them — a paragraph rejecting a corroborating letter by citing internal inconsistencies without naming the standard being applied. The matrix will show no pending cell for that paragraph. The paragraph will be marked dimension-coverage-complete even though the weighing framework was never applied to it.
Effect: Pre-applicability failure is a structural ceiling on framework registry quality that cannot be resolved by better pass scheduling or more disciplined session handoffs. For RPD decisions — which routinely embed multiple analytical issues within a single paragraph — this is a persistent limitation. The system must build in a schedule-revision mechanism at session boundaries when analysis reveals unanticipated framework-segment combinations not anticipated in the original applicability conditions.
